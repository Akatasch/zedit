xelib is a JavaScript wrapper around XEditLib.dll. XEditLib.dll is a dynamic-link library which provides an API to use the XEdit framework.

Namespaces

All members are top-level in xelib. This choice was made to simplify and shorten code which calls xelib functions.

Enumerations

Enumeration members in xelib are available as top level members. E.g. xelib.etRecord yields the ordinal value of the etRecord member of the elementTypes enumeration.

Enumeration objects contain an array of string values corresponding to the names of the enumeration members. E.g. xelib.defTypes[0] yields 'dtRecord', which is the name of the member at ordinal 0 in the defTypes enumeration.

Important Concepts

Elements

With xelib and the XEdit framework, everything in plugin files (including the files themselves) is an element. In zEdit's GUI, every row in the tree view and every row in the record view is an element. Elements can be files, groups, records, structs (a collection of a specific set of elements), arrays (a collection of some number of a single type of element), or values. All elements have a name, a type, and a value.

Paths

Several functions in xelib make use of paths. Paths are strings which describe how to reach an element from another element, or from the root. Parts of a path are separated by the backslash \, which needs to be escaped in javascript code (e.g. \\).

Files

You can access files from the root via the following keys:

  • Filename: The name of the file, including extension. E.g. 'Skyrim.esm'
  • Index: The index of the file in the files array. E.g. '[0]'. Note that the second file (index '[1]') is usually a hardcoded file containing records that are added by the game executable. (it will have the same load order as the main game ESM).

File Header

You can access a file's header via the following keys:

  • File Header: Simply use the key 'File Header'.
  • Index: Index '[0]' in a file will always return the file header.

Groups

You can access record groups in a file via the following keys:

  • Signature: The signature of top-level records in the group. E.g. 'Skyrim.esm\\WRLD'.
  • Name: The name of the group as displayed in the Tree View. E.g. 'Skyrim.esm\\Armor'.
  • Index: The index of the group. Indexes are based on the native order of groups in the ESP file. Note that index '[0]' returns the file header.

Records

You can access records from a file or group via the following keys:

  • Load order form ID: The load order form ID of the record. E.g. '0300ACE1'.
  • Editor ID: The Editor ID of the record. E.g. 'Skyrim.esm\\ArmorIronHelmet'. Note that this is considerably slower than accessing records by load order form ID unless the file's records have been sorted by Editor ID via xelib.SortEditorIDs.
  • Full Name: The FULL - Name of the record. E.g. 'Skyrim.esm\\"Iron Helmet"'. Note that this is considerably slower than accessing records by load order form ID unless the file's records have been sorted by Name via xelib.SortNames.
  • Index: The index of the record within a group. Indexes are based on the native order of records in the ESP file.

Fields (Record Elements)

You can access fields (elements in records) via the following keys:

  • Signature: The signature of the element. E.g. 'KWDA'.
  • Name: The name of the element displayed in the Record View. E.g. 'KWDA - Keywords'.
  • Index: The index of the element within the record. This should be used when accessing elements within arrays where each array item has the same name. E.g. 'Conditions\\[1]'
  • Pipe syntax: This is a special syntax which returns the element corresponding to the first path that exists. E.g. '[BODT|BOD2]'.

Handles

XEditLib.dll returns "handles" to elements. Handles are like pointers, but in the case of XEditLib they're integer IDs which correspond to entries in an internal store. Handles should be released with xelib.Release once your code is finished using them. There are several Utility functions which make working with handles easier.

Note: User scripts use a handle group so users don't have to worry about releasing handles created during script execution.

Note: Handle 0 is a special reserved handle. It emulates the "root" of the element tree when used with certain functions.
